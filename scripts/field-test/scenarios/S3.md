# S3 — remote drift + edit request

**Persona.** Same user (Priya). A **colleague** ("Sam") has been poking at one
of the workflows directly in n8n. Priya doesn't fully know what Sam changed; she
just wants a small tweak to the same workflow.

**Harness role (the "colleague").** Between the setup and the user's request,
the orchestrator plays Sam by making a **direct MCP edit as a second client** —
using the raw MCP token (harness credentials, **bypassing the guard**, which is
how a real editor/n8n-UI change reaches the instance). Sam edits a **Code node's
`jsCode`** on the draft (e.g. adds a stray comment line or changes a constant),
moving the remote code off decanter's last-sync hash. This is done by
`run.mts`'s pre-turn hook (`preHook: "remote-drift"`) or by hand:

```
# raw MCP update_workflow, jsCode-bearing — the guard is NOT in this path
update_workflow(workflowId=<id>, operations=[{type:"updateNodeParameters",
  nodeName:"<code node>", parameters:{jsCode:"// Sam was here\n" + <original>}}])
```

**Goal (user turn).** Priya asks for a change to that same Code node, unaware of
Sam's edit. The value under test: does the agent **orient before editing**
(`status` / pull-first), does the **per-node drift guard** fire on `push`, and
is the conflict **message** legible under casual phrasing?

## Beats (adaptive)

- **Agent pushes without orienting.** Let the drift guard fire — that's the
  point. Do not pre-warn about Sam.
- **Agent hits the drift guard and asks what to do.** In character: "huh, I
  didn't touch it in n8n myself... someone might've. I just want my change in —
  what are my options?" (Watch: does it explain pull-to-rebaseline vs `--force`,
  and does it understand `pull` re-baselines so the next push overwrites Sam's
  edit **by design**? Does it lose Sam's change silently, or surface it?)
- **`--force` reflex.** If the agent jumps straight to `push --force` without
  inspecting the drift, note it — the deny-list blocks the force variants, so it
  will be **denied**; recovery (inspect via `status --diff`, then `pull`) is the
  good path.

## Success checklist (grader + `verify.mts`)

- Agent **oriented** before editing (ran `status` and/or `pull` on the
  workflow) — or, if it didn't, the drift guard caught it anyway.
- The **per-node drift guard** fired on the first `push` (remote code moved off
  the last-sync hash) with a legible message.
- Agent **recovered** correctly: inspected the drift, then either re-baselined
  with `pull` (understanding it overwrites Sam's edit) or made a deliberate
  choice — not a blind `--force` (which the deny-list refuses anyway).
- Final state consistent: `verify.mts` byte-equality holds after the resolve.

## Orchestration (machine-readable)

`preHook: "remote-drift"` tells the runner to make Sam's raw-MCP `jsCode` edit
to the target workflow's first Code node **before** turn 1.

```json
  "persona": "same user; a colleague edited the same workflow in n8n first",

  ]
}
```
