# S2 — medium build via skills + the `mcp connect` guard

**Persona.** Same user (Priya), same sync dir, a **later session**. `init` has
already run, so `.mcp.json` exists and the blind agent's harness spawns the
stdio `mcp connect` guard at session start — this scenario actually exercises
it. The n8n skills pack is installed in the project.

**Goal.** Build a real multi-step flow: a **schedule trigger → Code (generate a
batch of synthetic order records) → IF (split by amount ≥ 100) → two Code
branches (tag "high" / "low") → Merge → Code (summarise counts)**. Structure is
n8n's job — created through the n8n MCP tools/skills **via the guard**; each
Code node's source rides the **designed** block→pull→seed loop. Finish by
proving it runs.

## The designed block→pull→seed loop (grade as SUCCESS, not recovery)

A Code node added over MCP **without** `jsCode` — the guard **blocks** `jsCode`
in `addNode`/`updateNodeParameters` — lands as an **empty** `code/<node>.js`
file. Because `liveMirror` is **on** (the honest default), a tracked workflow's
empty file materialises via the guard's **debounced background pull**; a
freshly `create_workflow_from_code`'d, still-untracked workflow needs a manual
`pull <id>` first. The agent then authors the file and **`push` seeds** the
source. This exact sequence is the product working as intended:

1. structure/add over MCP (guard forwards it) →
2. empty `code/` file appears (live-mirror pull, or manual `pull`) →
3. author the file locally →
4. `push` seeds the remote `jsCode`.

Mid-scenario "empty remote vs empty file" byte-equality is **legitimate** until
the first push. A guard **block** of a `jsCode` write is **expected**; only a
genuine stall (no recovery, or repeated `jsCode`-over-MCP retries) is a finding.

## Beats (adaptive)

- **Agent retries `jsCode` over MCP after a block.** Do **not** correct it —
  this is the finding signal. Log the retry count and whether it self-recovers
  to the file+push flow.
- **Agent can't get the empty file.** If it forgets a tracked workflow
  auto-mirrors, or the workflow is still untracked, it may hand-look for the
  file. Nudge only if fully stalled: "isn't there a pull step to bring the new
  node down?"
- **Change of mind (turn 2).** "for the summary at the end — just give me one
  field with the totals, like `{high: n, low: n}`, not separate items."
- **Shipping (turn 3).** "can you run it once so I can see it produces
  something sane?" — watch for `scenario create <wf> --scaffold` (synthetic
  pins, no captures exist yet) → `n8n-decanter test --scenario <slug>`, and/or
  offline `node run` fixtures. A `test --scenario` on synthetic pins proves
  **executability, not output correctness** — grade it as such.

## Success checklist (grader + `verify.mts`)

- Structure (trigger, IF, Merge, wiring) created via the **n8n MCP tools through
  the guard** — not by hand-editing `workflow.json`.
- Every Code node's source authored **in files + push**; the guard's stderr
  (`guard.log`) shows the **blocked `jsCode` writes** (this is the Plan 50
  authoring-skill evidence — did the skills' routing nudge bite?).
- block→pull→seed completed for each Code node; **no rogue `jsCode`** on the
  instance (`verify.mts` byte-equality holds after the final push).
- The flow ran once (`test --scenario` on synthetic pins and/or `node run`);
  agent understood synthetic pins ≠ correctness proof.
- `verify.mts` green for the whole workflow.

## Success checklist — evidence capture (Plan 50)

Explicitly answer, with `guard.log` excerpts: **did the skills' routing nudge
bite?** i.e. did the agent attempt `jsCode` over MCP (guard blocked, warn-line
emitted) before falling back to files, or did it go file-first unprompted? Count
attempts per Code node.
