# S1 — green field, simple (CLI-only, project already configured)

**Persona.** Priya, a solo operations analyst. Her n8n is running and this folder
is already set up to sync with it (she did that last week). She keeps her flows
in a git folder and wants one small workflow finished. Comfortable in a terminal,
not an n8n expert.

**Design decision (maintainer call 2026-07-24).** The **stage pre-runs
`n8n-decanter init`** (non-interactively, via the `--host`/`--token` flags) and
pre-installs the n8n skills pack, so the blind session **arrives at a configured
project**: `.env`, `.mcp.json` (the guard), `AGENTS.md`, `.claude/`,
`decanter.config.json`, `tsconfig.json`, `n8n-globals.d.ts` all exist. Setup is
**not** what this scenario measures — the *work* is. (Init's own UX is covered by
unit/e2e tests and round-1 findings 2 and 3; re-testing it here just burned turns.)

S1 stays **CLI-only against a stage-seeded skeleton**: a manual-trigger → **empty
Code node** workflow (`availableInMCP = true`), so S1 exercises discover → `pull`
→ author → verify → `push`. The MCP-guard *creation* path is S2's job.

**Goal.** Find and pull the skeleton workflow, write the Code node to dedupe and
normalise a pasted JSON list, confirm it works on the sample, then push. (It is a
MANUAL-trigger flow, so it is never published/activated — see the publish beat.)

## Beats (adaptive — send only if the condition fires)

- **Agent can't find the workflow.** `decanter.config.json` lists no workflows
  yet, so a bare `pull` has nothing to act on — it must discover (e.g.
  `list --remote`) and pull by name/id. If it stalls, nudge in character: "it's
  called 'Contact normalizer' in n8n." (Record the stall — discoverability signal.)
- **Agent authors the code but asks before touching the instance.** Observed
  2026-07-24: it finished + verified locally, then asked *"want me to `push` it
  now?"* — cautious-by-default, which is reasonable agent behaviour. The turn-1
  prompt now states the change must land in n8n, so this should not recur; if it
  still does, that IS a finding (the contract doesn't make file→push obvious).
  **Never** name the verb in a nudge — whether it knows to `push` is the test.
- **Agent asks how to verify before shipping.** Answer in character: "yeah I'd
  rather not flip it live until I've seen it do the right thing on a sample."
  (Watch whether it reaches for `preflight`, `node run`, `check`, or `test` —
  record which; do not name a verb for it.)
- **Change of mind (send as turn 2 regardless).** "oh — and drop any rows where
  `status` is `archived`, we don't care about those."
- **Agent tries to `publish`.** It will FAIL, legitimately: the skeleton's only
  trigger is a **manual** trigger, and n8n cannot *activate* a manual-trigger
  workflow (`publish_workflow: … has no trigger node`). That is correct n8n
  behaviour, not an agent error and not a scenario goal — a manual "paste JSON
  and run it" flow is never published. **Record how decanter surfaced it**
  (n8n's raw refusal, or a decanter-side explanation): a bare passthrough is a
  product-message finding worth logging.

## Success checklist (grader + `verify.mts`)

- The skeleton was **discovered and pulled** into `workflows/<slug>/` with an
  **empty** `code/` file for the Code node (born without `jsCode`). Record how it
  discovered it (`list --remote`, picker, the prompt's name).
- The Code node source was authored **in the file**, not via MCP; dedupe +
  normalise + (turn 2) drop-archived logic present.
- Some **local verification** ran before publish — `preflight`, `node run` with
  a fixture, `check`, or `test`. **Record which** (this is the graded
  preflight-adoption question).
- `push` reported a **draft** update (live unchanged). **No publish is expected** —
  n8n cannot activate a manual-trigger workflow; an attempt failing is correct.
- `verify.mts`: remote `jsCode` **byte-equals** the local `.js`; workflow.json
  placeholder intact; `.decanter.json` only via `decanter:` auto-commits.
